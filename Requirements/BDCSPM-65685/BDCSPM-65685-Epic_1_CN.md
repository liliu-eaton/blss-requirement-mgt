# BDCSPM-65685 — RDGC架构增强：连接稳定性、弹性恢复与远程诊断

## EPIC 1: RDGC-RDGS 连接稳定性、自愈与远程诊断

**BLSS 产品**: DCPM / 分布式IT性能管理 (Remote Data Gateway — RDGC & RDGS 组件)

**摘要**
增强RDGC架构，实现RDGC与RDGS之间稳定、可靠的连接，提供自愈和远程恢复能力，并支持终端用户通过界面收集诊断信息（日志、配置、系统状态），供BLSS技术支持进行高效的问题定位。

---

### 描述

- **问题陈述**: 当前RDGC-RDGS通信架构存在长期稳定性问题。RDGC与RDGS之间基于MQTT的连接频繁断开（参见相关缺陷：BDCSPM-66638 "所有RDGC与RDGS断开连接"、BDCSPM-66338 "RDGC持续报告MQTT断开告警"、BDCSPM-37494 "RDGC偶发断开连接"、BDCSPM-56990 "RDGC每五分钟断开一次"）。用于在线/离线检测的心跳机制过于复杂，导致RDGC在仍然发布数据的情况下被错误标记为离线。当断连发生时，故障排查困难：没有终端用户可自助收集诊断数据的手段，恢复需要人工介入，BLSS技术支持无法远程高效定位根因。

- **方案概述**:
  1. **连接稳定层**: 重新设计RDGC-RDGS通信机制使其更具弹性——简化心跳/保活逻辑，实现带指数退避的稳健MQTT重连策略，确保在线/离线状态准确反映实际数据发布状态。
  2. **自愈与远程恢复**: 实现当检测到连接异常时的自动恢复机制（如自动重连、服务重启编排、配置重新同步），无需人工干预。
  3. **诊断数据收集与下载**: 在Server Admin界面提供一个页面，管理员可触发诊断数据包生成（日志、RDGC/RDGS配置、连接状态、系统指标），并下载该数据包提交给BLSS技术支持。
  4. **连接健康监控**: 实时连接状态看板，含历史连接指标、重连事件时间线及持续断连模式的告警。

- **用户角色**: 本地IT/系统管理员、BLSS技术支持工程师、调试工程师、生命周期服务工程师

- **业务价值**:
  - 消除因RDGC断连问题导致的反复现场升级（RDG相关支持工单的首要类别）
  - 通过自助诊断将连接问题的平均解决时间(MTTR)从小时/天缩短到分钟
  - 通过远程恢复能力减少现场介入需求
  - 提升系统整体可靠性和客户对RDG架构的信心
  - 使BLSS技术支持无需VPN访问客户现场即可进行根因分析

- **关键工作流**:
  1. RDGC使用简化的、可靠的保活机制建立与RDGS的MQTT连接
  2. 连接丢失时，RDGC使用指数退避策略自动尝试重连
  3. 超过可配置重试阈值后，RDGC触发自愈流程（服务重启、配置重载）
  4. 系统基于实际连接状态+数据发布状态准确报告RDGC在线/离线状态
  5. 管理员导航至 Server Admin → RDG管理 → 诊断页面
  6. 管理员触发"生成诊断包"，收集日志、配置、连接历史、系统状态
  7. 管理员下载诊断包(.zip)并提供给BLSS技术支持进行分析
  8. BLSS技术支持使用诊断包定位根因，无需远程接入

- **跨产品关联**: RDGC连接稳定性影响所有依赖远程站点数据采集的BLSS产品（DCPM用于远程设施监控，分布式IT用于边缘站点管理）。诊断包下载使用各产品共享的ServerAdmin UI框架。

- **UX/UI参考**: `[待定 — RDG诊断页面和连接健康看板的UX原型]`

- **本地部署运维影响**:
  - 安装/升级影响：RDGC新增诊断服务组件；RDGS连接管理模块更新。升级必须保留现有RDGC注册信息和连接配置。
  - 资源影响：RDGC需要额外磁盘空间存储诊断日志（可配置，默认500MB滚动）。连接健康监控的CPU/RAM开销极小。
  - 网络影响：无需新端口——使用现有MQTT通道。诊断包下载通过现有ServerAdmin HTTPS端口。
  - 备份影响：连接配置和健康历史纳入RDGC快照/备份范围。

### STORY S1: 简化RDGC心跳/保活机制

**父Epic**: BDCSPM-65685-Epic_1 — RDGC-RDGS连接稳定性、自愈与远程诊断
**BLSS产品**: RDG (RDGC组件)

**描述**
作为平台工程师，
我希望RDGC与RDGS之间有一个简化且可靠的心跳机制，
以便系统能够快速准确地检测RDGC连接状态，避免因复杂的payload构造导致误报离线事件。

**上下文与说明**:
- 当前心跳payload构造过于复杂，在某些场景下会导致心跳缺失（参见BDCSPM-65685描述）
- RDGC已经通过MQTT发布数据；心跳应利用相同通道，以最小开销实现
- 简化心跳 = 轻量消息（仅包含时间戳 + RDGC ID），固定间隔发送
- 必须在滚动升级期间保持与现有RDGS版本的向后兼容

**验收标准**

AC1:
  前置条件: RDGC已连接到RDGS并正常发布数据
  当: 心跳间隔定时器触发（默认30秒，可配置）
  则: 在1秒内向MQTT心跳主题发布一条轻量心跳消息（< 256字节）

AC2:
  前置条件: RDGC处于高数据发布负载（100+设备同时上报）
  当: 心跳间隔定时器触发
  则: 心跳仍按时发送（在间隔的±5秒内），不被数据发布阻塞

AC3:
  前置条件: RDGC心跳配置为默认值
  当: 系统连续运行72小时
  则: RDGS日志中观察到零次心跳缺失

AC4（异常场景）:
  前置条件: RDGC到RDGS的网络连接被物理切断
  当: 心跳间隔触发
  则: RDGC本地记录心跳失败日志（含时间戳和错误原因）；触发重连流程

**任务分解**
- [ ] 任务1: 设计简化心跳消息格式（RDGC ID + 时间戳 + 序列号）
- [ ] 任务2: 重构心跳发送器，使其与数据发布管道解耦（独立线程/任务）
- [ ] 任务3: 心跳间隔通过RDGC配置文件可配置（默认30秒，最小10秒，最大120秒）
- [ ] 任务4: 更新RDGS心跳接收器以接受新的简化格式（向后兼容）
- [ ] 任务5: 添加心跳发送/接收指标（次数、延迟、缺失数）到内部指标系统
- [ ] 任务6: 编写心跳发送器在各种负载条件下的单元测试
- [ ] 任务7: 编写集成测试：72小时稳定性测试，并发数据发布

### STORY S2: 实现带指数退避的稳健MQTT重连

**父Epic**: BDCSPM-65685-Epic_1
**BLSS产品**: RDG (RDGC组件)

**描述**
作为系统管理员，
我希望RDGC在网络中断后能使用智能重试逻辑自动重连到RDGS，
以便临时网络问题不需要人工干预即可恢复数据网关连接。

**上下文与说明**:
- 当前行为：RDGC可能保持断连状态或进入快速重连循环（雷群效应）
- 新行为：指数退避 + 抖动（初始2秒，最大5分钟，抖动±20%）
- 必须处理正常断开（MQTT DISCONNECT）和非正常断开（TCP超时）两种情况
- 重连成功后，必须重新订阅所有命令主题并恢复数据发布
- 重连尝试和结果必须记录日志供诊断使用

**验收标准**

AC1:
  前置条件: RDGC已连接到RDGS
  当: 网络中断 < 30秒
  则: RDGC在60秒内自动重连并恢复数据发布，断连期间数据无丢失（已缓冲）

AC2:
  前置条件: RDGC已断连 > 5分钟
  当: 网络恢复
  则: RDGC按退避计划重连，在网络可用后30秒内恢复连接

AC3:
  前置条件: 50+个RDGC同时与一个RDGS断开
  当: 网络恢复
  则: RDGC使用随机抖动重连，防止雷群效应；重连风暴期间RDGS CPU保持 < 80%

AC4:
  前置条件: RDGC处于重连退避状态
  当: 管理员在Server Admin中查看RDGC状态
  则: 状态显示"重连中"，含下次重试时间和尝试次数

**任务分解**
- [ ] 任务1: 实现指数退避算法（基数2秒，乘数2倍，最大5分钟，抖动±20%）
- [ ] 任务2: 实现断连期间的本地消息缓冲（可配置大小，默认1000条消息或10MB）
- [ ] 任务3: 实现重连状态机（已连接 → 已断开 → 重连中 → 已连接）
- [ ] 任务4: 添加重连指标（尝试次数、总断连时长、缓冲区利用率）
- [ ] 任务5: 测试100个并发RDGC的雷群场景
- [ ] 任务6: 实现重连后的优雅缓冲区刷新（限速以避免RDGS过载）

### STORY S3: 精确在线/离线状态检测

**父Epic**: BDCSPM-65685-Epic_1
**BLSS产品**: RDG (RDGS组件)

**描述**
作为系统管理员，
我希望系统中显示的RDGC在线/离线状态能准确反映真实的连接和数据发布状态，
以便我可以信任监控看板并做出明智的运维决策。

**上下文与说明**:
- 当前问题：RDGC可能在仍然活跃发布数据的情况下被标记为"离线"（误报）
- 根因：仅依赖心跳检测会遗漏心跳失败但数据流正常的情况
- 新方案：双信号检测——连接状态（MQTT会话）AND 数据流活跃度（最后消息时间戳）
- RDGC"在线" = MQTT会话活跃 AND（在阈值内收到心跳 OR 在阈值内收到数据消息）
- RDGC"离线" = MQTT会话断开 AND 在2×心跳间隔内无数据消息接收

**验收标准**

AC1:
  前置条件: RDGC有活跃的MQTT会话且正在发布数据
  当: 心跳缺失（如由于临时处理延迟）
  则: RDGC状态保持"在线"，因为数据消息仍在接收中

AC2:
  前置条件: RDGC MQTT会话断开
  当: 2×心跳间隔（默认60秒）内无数据消息接收
  则: RDGC状态变为"离线"，显示最后已知活跃时间戳

AC3:
  前置条件: RDGC在离线后重新连接
  当: RDGS收到第一条心跳或数据消息
  则: RDGC状态在5秒内变为"在线"

AC4:
  前置条件: 系统有100个已注册RDGC
  当: 所有RDGC正常运行24小时
  则: 零次误报"离线"事件产生

**任务分解**
- [ ] 任务1: 在RDGS上实现双信号状态检测器（MQTT会话状态 + 最后消息时间戳）
- [ ] 任务2: 在RDGS上添加每个RDGC的"最后数据消息接收"时间戳追踪
- [ ] 任务3: 实现可配置的离线检测阈值（默认：2×心跳间隔）
- [ ] 任务4: 更新状态API，返回连接状态及置信度和证据
- [ ] 任务5: 更新Server Admin UI状态显示以反映新的状态逻辑
- [ ] 任务6: 编写集成测试：模拟心跳缺失但数据正常流动（验证无误报离线）

- 所有恢复操作必须记录日志供审计和事后分析
- 恢复期间数据已缓冲；最大缓冲限制防止磁盘耗尽

**验收标准**

AC1:

  则: RDGC按顺序尝试恢复步骤（MQTT客户端重置 → 配置重载 → 服务重启）

AC2:
  前置条件: 自愈步骤1（MQTT客户端重置）成功
  当: 连接恢复
  则: 30秒内恢复正常运行；恢复事件记录为"在步骤1解决"

AC3:
  前置条件: 所有自愈步骤失败
  当: 最大恢复尝试次数耗尽（3个周期）
  则: RDGC进入"恢复失败"状态，记录详细失败信息，RDGS显示告警通知管理员

AC4:
  前置条件: 自愈触发了服务重启
  当: 重启完成
  则: 所有之前缓冲的数据消息按顺序刷新到RDGS；无数据丢失
